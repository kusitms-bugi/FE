# Feature Specification: PR Merge Auto Release

**Feature Branch**: `001-auto-release-versioning`  
**Created**: 2026-03-27  
**Status**: Draft  
**Input**: User description: "배포 흐름을 안정화하고 싶어 피알 머지하면 바로 버전업해서 배포되게"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 머지 후 자동 릴리즈 시작 (Priority: P1)

릴리즈 담당자는 검증이 끝난 PR을 기본 브랜치에 머지한 뒤 별도의 수동 태그 생성이나
버전 수정 없이 즉시 릴리즈가 시작되길 원한다. 이 흐름은 정식 배포를 위한 단일 진입점이
되어야 하며, 누가 머지하더라도 동일하게 동작해야 한다.

**Why this priority**: 현재 가장 큰 문제는 배포 시작 조건이 수동 작업에 의존해 누락과
불일치가 발생할 수 있다는 점이다. 머지 직후 자동으로 릴리즈를 시작하는 것이 가장 큰
안정화 효과를 만든다.

**Independent Test**: 승인된 PR을 기본 브랜치에 머지했을 때, 시스템이 자동으로 다음
버전을 계산하고 릴리즈 절차를 시작하며, 담당자가 별도 버전 입력이나 태그 작업을 하지
않아도 배포가 진행되면 된다.

**Acceptance Scenarios**:

1. **Given** 배포 가능한 PR이 모든 필수 검증을 통과한 상태이고 기본 브랜치로 머지되면,
   **When** 자동 배포 흐름이 시작되면, **Then** 시스템은 즉시 다음 릴리즈 버전을
   결정하고 배포 절차를 시작한다.
2. **Given** 같은 변경에 대해 이미 릴리즈가 시작된 기록이 있으면, **When** 중복
   트리거가 발생하면, **Then** 시스템은 같은 버전으로 중복 배포를 만들지 않는다.

---

### User Story 2 - 일관된 버전 규칙 보장 (Priority: P2)

릴리즈 담당자는 버전이 사람마다 다르게 올라가지 않길 원한다. 변경 성격에 따라 버전이
일관되게 결정되고, 릴리즈 산출물과 사용자에게 노출되는 버전이 서로 일치해야 한다.

**Why this priority**: 자동 배포가 되더라도 버전 규칙이 흔들리면 업데이트 추적, 롤백,
지원 대응이 모두 어려워진다.

**Independent Test**: 서로 다른 성격의 머지 결과를 기준으로 버전 증가 유형이 예측 가능하게
결정되고, 릴리즈 기록과 앱 버전이 동일하게 보이면 된다.

**Acceptance Scenarios**:

1. **Given** 배포 대상 변경이 이전 정식 배포 이후 누적되어 있으면, **When** 자동
   릴리즈가 버전을 계산하면, **Then** 버전 증가는 사전에 정한 규칙에 따라 한 가지
   방식으로만 결정된다.
2. **Given** 새 릴리즈가 생성되면, **When** 사용자가 릴리즈 정보와 앱 업데이트 정보를
   확인하면, **Then** 모든 위치에서 동일한 버전 문자열을 확인할 수 있다.

---

### User Story 3 - 실패 지점과 롤백 판단 가시화 (Priority: P3)

운영 담당자는 자동 릴리즈가 실패했을 때 어느 단계에서 멈췄는지 바로 알고 싶다. 또한
치명적인 실패가 생기면 마지막 안정 버전으로 되돌릴 수 있는 판단 근거를 확보하고 싶다.

**Why this priority**: 자동화는 성공보다 실패 시의 대응 속도가 더 중요하다. 실패 상태가
불명확하면 자동화가 오히려 배포 위험을 키운다.

**Independent Test**: 배포 흐름 중 한 단계를 실패시켰을 때 실패 단계, 영향 범위, 다음
   조치가 즉시 확인되고, 성공한 마지막 안정 릴리즈를 기준으로 롤백 판단이 가능하면 된다.

**Acceptance Scenarios**:

1. **Given** 자동 릴리즈가 버전 산정 이후 어느 단계에서 실패하면, **When** 담당자가
   배포 결과를 확인하면, **Then** 실패 단계와 원인 요약, 후속 조치가 한 곳에서 보여야
   한다.
2. **Given** 릴리즈 산출물 일부만 완료된 상태에서 실패하면, **When** 담당자가 운영
   판단을 내리면, **Then** 배포 완료 여부와 롤백 필요 여부를 명확히 구분할 수 있어야
   한다.

### Edge Cases

- 같은 시점에 여러 PR이 연속으로 머지될 때 버전 계산과 릴리즈 순서가 충돌하지 않아야 한다.
- 릴리즈 시작 후 일부 플랫폼만 성공한 경우 전체 릴리즈 상태를 부분 성공으로 숨기지 않아야 한다.
- 버전 증가 규칙을 판단할 메타데이터가 누락된 PR은 임의 배포하지 않고 명확한 실패로 처리해야 한다.
- 릴리즈 노트 생성에는 성공했지만 배포 산출물 업로드가 실패한 경우, 사용자에게 배포 완료로
  표시하지 않아야 한다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 기본 브랜치에 머지된 배포 대상 변경을 감지하면 자동으로 릴리즈 흐름을 시작해야 한다.
- **FR-002**: 시스템은 릴리즈 시작 시 다음 정식 버전을 자동 산정해야 하며, 동일 변경에 대해 중복 버전을 만들면 안 된다.
- **FR-003**: 시스템은 버전 산정 결과를 배포 산출물, 릴리즈 기록, 사용자 업데이트 정보에 일관되게 반영해야 한다.
- **FR-004**: 시스템은 머지된 PR의 `release:major`, `release:minor`, `release:patch` label 중 정확히 하나를 기준으로 버전 증가 수준을 판정해야 한다.
- **FR-005**: 시스템은 버전 판정 label이 없거나 둘 이상 존재하면 추정 배포를 진행하지 않고 명시적으로 실패 처리해야 한다.
- **FR-006**: 시스템은 릴리즈 시작, 버전 결정, 태그 생성, 플랫폼별 배포, 메타데이터 업로드의 각 단계를 추적 가능하게 기록해야 한다.
- **FR-007**: 시스템은 플랫폼별 배포 결과를 통합해 릴리즈 성공, 부분 실패, 전체 실패를 구분해 표시해야 한다.
- **FR-008**: 시스템은 실패 시 담당자가 바로 확인할 수 있는 요약 정보와 후속 조치 지침을 제공해야 한다.
- **FR-009**: 시스템은 마지막 성공 릴리즈를 기준으로 롤백 판단에 필요한 버전, 시각, 산출물 상태를 보존해야 한다.
- **FR-010**: 시스템은 `main` merge 기반 자동 릴리즈를 기본 경로로 사용해야 하며, `workflow_dispatch` 기반 수동 릴리즈는 자동 경로 장애 복구 또는 운영 승인 상황에서만 예외적으로 사용되어야 한다.
- **FR-011**: 시스템은 자동 릴리즈 시작 전에 해당 커밋 SHA에 대해 필수 GitHub Actions 체크(`lint`, `typecheck`, `build`)가 모두 성공 상태인지 검증해야 하며, 하나라도 성공이 아니면 릴리즈를 시작하면 안 된다.
- **FR-012**: 시스템은 릴리즈 결과를 GitHub Release 본문과 저장소 운영 문서에 반영할 수 있어야 하며, 자동 릴리즈 정책 변경은 ADR에 기록되어야 한다.

### Constitution Alignment *(mandatory for implementation planning)*

- **CA-001**: 이 기능은 여러 프로세스를 동시에 변경하지 않으며, 저장소의 배포 자동화 경로와 앱 버전 노출 지점을 함께 다룬다.
- **CA-002**: renderer에 새 시스템 권한을 추가하지 않는다. 필요 시 배포 상태 노출은 기존 안전한 계약 경로 안에서만 읽기 전용으로 다뤄야 한다.
- **CA-003**: 앱 런타임 코드 변경이 필요하면 릴리즈 정보 표시 로직은 적절한 renderer 레이어에 두고, 배포 제어 로직은 renderer 밖에서 관리해야 한다.
- **CA-004**: 릴리즈 진행 상태가 앱 UI에 노출되더라도 일시적 표시 상태와 영속적인 배포 기록의 소유 경계를 분리해야 한다.
- **CA-005**: 이 기능은 release flow 변경에 해당하므로 자동 버전 산정, 태그 생성, 산출물 업로드, 실패 로그, 롤백 판단 기준에 대한 회귀 검증과 운영 로그가 필요하다.

### Key Entities *(include if feature involves data)*

- **Release Candidate**: 다음 정식 배포로 확정되기 전의 머지 결과 묶음. 기준 브랜치 커밋, 후보 버전, 포함 변경 목록을 가진다.
- **Version Decision**: 후보 변경에 대해 결정된 버전 증가 결과. 이전 버전, 다음 버전, 증가 수준, 판정 근거를 가진다.
- **Release Execution**: 실제 배포 실행 기록. 시작 시각, 단계별 상태, 플랫폼별 결과, 최종 결론을 가진다.
- **Release Outcome**: 사용자에게 공개된 배포 결과. 버전, 산출물 완성 여부, 업데이트 메타데이터 상태, 롤백 필요 여부를 가진다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 승인된 변경이 기본 브랜치에 머지된 후 5분 이내에 새 릴리즈 버전이 결정되고 배포가 시작된다.
- **SC-002**: 정식 배포의 95% 이상이 사람의 수동 버전 수정이나 수동 태그 생성 없이 완료된다.
- **SC-003**: 릴리즈 실패 건의 100%에서 실패 단계와 원인 요약을 10분 이내에 확인할 수 있다.
- **SC-004**: 릴리즈 버전 불일치로 인한 배포 재실행 또는 운영 수정 건수가 한 분기 내 0건이 된다.

## Assumptions

- 기본 브랜치 머지는 이미 필수 품질 검증을 통과한 변경만 허용된다고 가정한다.
- 현재 태그 기반 배포 흐름은 유지하고, 이 기능은 태그를 자동 생성하는 상위 자동화로 동작한다고 가정한다.
- 버전 판정 메타데이터는 PR label을 기준으로 하며 허용 값은 `release:major`, `release:minor`, `release:patch`뿐이라고 가정한다.
- 릴리즈 대상 PR에는 허용 label이 정확히 하나만 존재해야 한다고 가정한다.
- 운영팀은 마지막 안정 릴리즈를 기준으로 롤백 판단을 내리며, 롤백 자체의 실행 절차는 별도 운영 문서에서 관리한다고 가정한다.

## Documentation & Release Impact

- **ADR Needed**: Yes - 자동 버전 판정 규칙, 자동 태그 생성 정책, 실패 시 수동 개입 경계를 남겨야 한다.
- **Analytics/Event Schema Impact**: None for end-user product analytics. Operational release event logging is required.
- **Release Note Impact**: PR 머지 후 자동 릴리즈 시작, PR label 기반 버전 규칙 통일, 실패 가시화 개선 내용을 포함해야 한다.
- **Rollback Concern**: 자동 버전 산정이 잘못되거나 부분 배포 상태가 잘못 표시되면 잘못된 태그와 업데이트 메타데이터가 배포될 수 있다.
